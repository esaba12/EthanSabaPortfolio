# Phase 1 — UI Design Contract

> Visual and interaction contract for the "Engineering Blueprint" design system and Home page rebuild. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Revision Note (v2 — 2026-07-22)

**This is a revision of the v1 spec, not a new document.** v1 was implemented and approved on tokens, then rejected live by the user as "barely changed." Verbatim feedback (see PROJECT.md "Phase 1 v1 checkpoint feedback"): the token/motif system (colors, fonts, corner brackets, mono labels, the one hero line-wipe) was built correctly, but the underlying DOM layout was left untouched — same narrow `container mx-auto` centered column end to end, same plain bordered 2-column Skills `Panel`, same 3-card grid — so it read as a re-skin, not a redesign.

**What changed in this revision:** the **Layout & Composition** section (new) and the **Motion System** choreography contract (rewritten) below. Design System, Spacing Scale, Typography (except the Display row), Color, Grid & Canvas Pattern, and Registry Safety are carried forward unchanged — those were not the source of the complaint and remain locked. The Component Inventory gains new layout-carrying primitives; existing components (`Panel`, `CoordinateLabel`, `Badge`, `ProjectsGrid`, `WaveUnderline`) keep their APIs.

**Status reset to `draft`** — this revision must go back through the checker before Home page is re-approved. Checker Sign-Off below is marked PENDING RE-REVIEW, not carried forward from v1.

**Forward guidance for Phases 2–5:** the Layout & Composition section is written generally, not Home-specific, precisely so About/Projects/Creative/Contact UI-SPECs can cite it as precedent. **Full-bleed and asymmetric section composition is now the sitewide norm, not an exception requiring special justification.** Any future phase whose page-level layout is a single centered `container mx-auto` column stacked top-to-bottom, with no full-bleed or asymmetric moment, fails this contract by default and should be treated the same way v1 Home was — a re-skin, not a redesign.

**Aesthetic direction (explicit commitment, unchanged):** Retro-futuristic technical / brutalist-precision hybrid — "Engineering Blueprint." Charcoal canvas, drafting-grid, annotation typography, one committed signal-amber accent, cinematic GSAP-driven reveals. This is NOT soft-SaaS, NOT glassmorphism, NOT purple-gradient, NOT rounded-card-everywhere. **v2 addendum: it is also NOT a single vertically-stacked centered column with the motifs painted on top.** The blueprint metaphor implies drafting-table composition — offset panels, instrument rails, coordinate-anchored elements breaking a strict center line — not a brochure layout wearing blueprint colors.

## Spacing Scale

Unchanged from v1 — still governs everything, including the new full-bleed edge padding introduced in Layout & Composition below (edge padding must be drawn from this scale, not arbitrary values).

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, coordinate-label internal padding |
| sm | 8px | Compact element spacing, corner-bracket inset from panel edge |
| md | 16px | Default element spacing, badge/tag padding |
| lg | 24px | Card/panel internal padding, section label spacing |
| xl | 32px | Layout gaps between stacked blocks, blueprint grid cell size |
| 2xl | 48px | Major section breaks (Hero → Skills → Projects) |
| 3xl | 64px | Page-level top/bottom spacing, full-bleed section edge padding on desktop |

Exceptions (unchanged):
- Icon-only touch targets = 44px minimum hit area, not on the 8pt scale (accessibility floor).
- Corner-bracket arm length = 24px (`lg`), bracket stroke inset = 8px (`sm`) — sitewide.
- Blueprint grid line pitch = 32px (`xl`) desktop, 24px (`lg`) mobile.
- **New (v2):** full-bleed section edge padding (replacing `container`'s `px-4 sm:px-6 lg:px-8`) = `px-6` mobile (24px, `lg` token) / `px-12` tablet (48px, `2xl` token) / `px-16` desktop (64px, `3xl` token). Still 8pt-scale values, just applied as true viewport-edge padding instead of centering inside a max-width column.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 (Geist Sans Regular) | 1.5 |
| Label | 13px | 400 (Geist Mono Regular) | 1.4 |
| Heading | 28px (desktop) / 22px (mobile, <640px) | 700 (Chakra Petch Bold) | 1.2 |
| Display | **140px (desktop, ≥1024px) / 88px (tablet, 640–1023px) / 56px (mobile, <640px)** | 700 (Chakra Petch Bold) | 0.95 |

**v2 change:** Display size escalated from the v1 cap (64px desktop / 40px mobile) to 140px / 88px / 56px. This is the direct fix for feedback item 1 ("hero should fill the viewport at a much larger type scale"). Still only two weight values sitewide: **400** (Geist Sans body, Geist Mono labels) and **700** (Chakra Petch display + heading) — no third weight introduced.

**Exception to the "two explicit breakpoints" rule (v1 Typography note):** Display is now the one role that gets **three** explicit Tailwind breakpoint values (`text-[56px] sm:text-[88px] lg:text-[140px]`) instead of two, because it now spans a much wider practical range and must remain legible/wrap sensibly at tablet width, not just jump from mobile to desktop. All other roles (Body, Label, Heading) keep the two-breakpoint pattern from v1 — this is a scoped exception for Display only, not a reopening of the general rule.

**Reveal implication:** at 140px, the existing hero sentence ("Hi. I'm Ethan, and this is the work that I'm proud of.") will wrap across roughly 4–6 lines on desktop instead of 1–2. This is intentional and works with the existing `SplitText` per-line mask reveal already implemented (`type: 'lines', mask: 'lines'`) — more lines means a more dramatic staggered cascade, not a bug to fix by shrinking type back down.

Other notes (unchanged from v1):
- Label role: uppercase + `tracking-wide` (0.04em), used for `CoordinateLabel`, Badge chips, ProjectsGrid tag chips and filter buttons.
- Body line-height 1.5 sitewide (already applied in `globals.css`).
- Decorative/oversized numerals introduced in Layout & Composition below (the hero "ghost index" watermark) reuse the **Display** typeface (Chakra Petch Bold) at an ad-hoc larger size specific to that one element — this is a one-off decorative treatment, not a new typographic role, and does not count against the type-scale limit.

Unchanged from v1 — the persistent texture background. Distinct from the new macro-layout grid system below (that's page composition; this is the fixed decorative texture).

- Fixed-position full-viewport `<div>` behind all content, one instance mounted once in `layout.tsx`.
- Line color `rgba(255,255,255,0.04)` on `#0a0a0a`, never exceeds 0.06 opacity.

## Layout & Composition

**New section (v2).** This is the core fix for the "barely changed" feedback. It governs macro page composition — how content is arranged across the viewport — as distinct from the token-level Grid & Canvas Pattern above. Written generally so Phases 2–5 can cite it as precedent; the specific tasks below (hero, Skills rail) are Home-page scope only.

### Sitewide composition rule (applies to all future phases, not just Home)

Replace the single-tier `.container` (mx-auto, max-width capped at 1536px, `px-4 sm:px-6 lg:px-8`) as the *only* layout primitive with a **two-tier canvas system**:

| Tier | Purpose | Behavior |
|------|---------|----------|
| `canvas-full` (new utility) | Full-bleed sections — hero, section dividers, any instrument-rail/gallery-style content | No max-width cap, viewport-edge padding only: `px-6 sm:px-12 lg:px-16` (see Spacing Scale v2 exception above). Content may run edge-to-edge or be deliberately offset within it (left-aligned, right-aligned, asymmetric column split) — never auto-centered by default. |
| `canvas-content` (rework of existing `.container`) | True reading-width content — body paragraphs, the Real Estate blurb, project card copy | Keeps the existing max-width scale (640/768/1024/1280/1536px breakpoints) and centering. Reserved for content that genuinely needs a constrained reading measure, not used as the default page wrapper. |

**Rule going forward:** a page (or a phase's UI-SPEC) that wraps its entire content top-to-bottom in `canvas-content` alone, with zero `canvas-full` sections, does not meet this contract. **Every page must have at least one full-bleed or asymmetric moment.** This is the literal fix for feedback item 4 and is deliberately generalized so it's the default expectation for Phases 2–5, not a one-time Home exception.

**Asymmetric composition primitive:** for sections that need internal structure without full centering, use a 12-column CSS grid (`grid grid-cols-12 gap-6`) and place content in deliberately unequal spans (e.g. `col-span-8` + `col-span-4`, or content starting at `col-start-2` rather than centered) rather than defaulting to `flex justify-center items-center` or `mx-auto`. Centering is one option among several now, not the default.

### Home page hero (fixes feedback item 1 + 4)

- **Full-viewport section:** hero wrapper is `min-h-screen` (already true today) but now uses `canvas-full`, not `canvas-content` nested inside `canvas-content` (the current triple-nested `.container` divs in `page.tsx` are removed).
- **Not centered:** headline block is left-aligned within the full-bleed canvas, occupying roughly the left 8 of 12 grid columns on desktop (`col-span-8` or equivalent), not `text-center` / `mx-auto`. Sub-line and `CoordinateLabel` follow the same left alignment, directly under the headline — no more `text-center` treatment anywhere in the hero.
- **Right-column instrument readout:** the remaining ~4 columns on desktop (collapses to stacked-below on mobile, not hidden) host a vertical technical readout: the `[ 00 ] INTRO` `CoordinateLabel` promoted to sit here instead of stacked above the headline, plus 1–2 additional coordinate-style annotations (e.g. a mock `LAT/LNG`-style readout or a `STATUS: ONLINE` style line — copy TBD by executor within the existing mono/amber motif, not new "content" per Copywriting Contract's content-lock). This column is what visually breaks the centered-column read even before scroll starts.
- **Oversized ghost numeral (decorative, new element):** a large `00` or `[ EST. 2026 ]`-style numeral/annotation rendered in Chakra Petch Bold at a size well beyond the Display scale (e.g. 320–420px), positioned bleeding off the right or bottom edge of the hero section, white at 4–6% opacity (same visual weight class as the blueprint grid texture — never competes with the headline). This is the parallax element referenced in Motion System below.

### Skills section (fixes feedback item 2)

Replace the v1 "one `Panel` wrapping a 2-column div" shape entirely. New direction: **technical instrument rail** — individual bracketed tiles, not one shared box.

- **Programming skills → 4 individual `SkillTile` panels** (new component, see Component Inventory), one per language (C++/C, Python, JavaScript, Java). Each tile is its own small `Panel` instance (corner brackets, hairline border) containing: a `CoordinateLabel` index (`[ 01 ]`–`[ 04 ]`), the skill name set in Heading role, and one short mono "spec line" (e.g. a made-up-but-plausible technical annotation like a version/proficiency readout — executor's discretion within the mono/amber motif, does not violate content lock since it's motif chrome, not prose).
- **Layout:** on desktop, the 4 `SkillTile`s sit in a row spanning 8 of 12 columns (`col-span-2` each), full-bleed edge-to-edge inside `canvas-full` — not centered, not capped at reading width. On mobile, they become a horizontal scroll-snap rail (`overflow-x-auto snap-x`, each tile `snap-start`) rather than stacking vertically, so the "instrument rail" character survives on small screens instead of collapsing into another plain stack.
- **Real Estate → one wider, taller `Panel`**, occupying the remaining 4 columns on desktop, offset to the right of the tile row (not below it, not the same width/height as the tiles) — this asymmetry (4 equal small tiles + 1 larger unequal panel) is the structural fix: it reads as a dashboard, not a bordered box with two columns inside it.
- Both the tile row and the Real Estate panel sit inside the same `canvas-full` section so they can run genuinely edge-to-edge on wide viewports.

### Featured Projects section

No structural change required to meet the escalation bar (3-card grid is acceptable card-grid content, unlike the Skills box which was flagged specifically). Optional, executor's discretion: shift `ProjectsGrid`'s wrapping section into `canvas-full` with `canvas-content`-width inner padding, so its edge padding is consistent with the new hero/Skills sections rather than the old `.container` value — but this is polish, not a requirement.

## Motion System

Rewritten choreography contract (v2). Stack, integration pattern, and the existing hero line-wipe mechanism are **unchanged and already correctly implemented** — do not rebuild them. What's added below are three new choreography requirements that satisfy feedback item 3 ("richer scroll choreography ... something that makes scrolling itself feel choreographed").

**Stack (unchanged, already installed and wired):**
- `gsap` + `ScrollTrigger`, `@gsap/react` (`useGSAP()`), `lenis` (`lenis/react`), `framer-motion` for micro-interactions. No new packages required for v2.
- `<ReactLenis root>` in `layout.tsx`, `ScrollTrigger` registered once at module scope in `motion-utils`, Lenis synced to `gsap.ticker` — all already in place per the existing `page.tsx`/`motion-utils.ts`, keep as-is.
- Every effect still gated by `prefersReducedMotion()` with an instant fallback to end-state — this pattern must extend to all three new effects below, not just the existing hero reveal.

### Choreography contract (v2 — supersedes v1's "one big moment" framing)

v1's single scroll-stopping moment is necessary but was not sufficient — it front-loaded all the ambition into the first 1.2s of the page and left everything after it inert. v2 keeps that moment and adds ambition across the scroll, per feedback item 3.

1. **Hero reveal + pin (escalates the existing moment, does not replace it).** Keep the existing `SplitText` line-wipe + `WaveUnderline` draw exactly as implemented. Add: wrap the hero in a `ScrollTrigger` with `pin: true` for a short scroll distance (roughly 60–100vh of scroll, `end: '+=80%'` or similar) immediately after the initial reveal completes. While pinned, scrub a secondary state change — e.g. the headline scales down slightly (`scale: 1 → 0.9`) and/or the right-column instrument readout content shifts — so the first scroll input feels *captured and choreographed* rather than the page simply scrolling the hero away. Release the pin into the Skills section.
2. **Scroll-scrubbed parallax on the ghost numeral (new, required).** The oversized decorative numeral/annotation introduced in Layout & Composition (hero right/bottom edge) moves at a different vertical speed than the foreground content via a scrubbed `ScrollTrigger` (`scrub: true`, no pin) — e.g. translateY at roughly 0.4–0.6x the scroll speed of the hero content, so it visibly drifts slower and produces a depth cue as the user scrolls past the hero. This is a *different* element from the static `BlueprintGrid` texture (which still does not animate, per Grid & Canvas Pattern) — do not parallax the grid itself, only this one decorative numeral.
3. **Staggered reveal on the Skills Instrument Rail (replaces the v1 single fade+8px block for this section only).** Each `SkillTile` (and the Real Estate panel) animates in individually via a staggered `ScrollTrigger` timeline as the rail scrolls into view — opacity 0→1 and a small scale-in (0.92→1), staggered ~0.08–0.12s per tile, `power2.out`, triggered once at `top 80%` of the rail container. This replaces the "same fade+8px-rise treatment for both Skills and Projects" instruction from v1 for the Skills section specifically.
4. **Featured Projects (unchanged from v1).** Keeps the simple fade+8px-rise reveal (opacity 0→1, y 8px→0, `power2.out`, ~0.4s, `top 80%`) — this section was not flagged and stays restrained by design, so the whole page doesn't become uniformly maximal.
5. **Micro-interactions (unchanged from v1):** Badge focus/hover, nav pill active-state, ProjectsGrid card hover-scale — stay on framer-motion, existing durations.
6. **Do not animate:** the `BlueprintGrid` texture background, footer, or nav on scroll — unchanged. The new ghost-numeral parallax (item 2) and hero pin (item 1) are the only additions to what animates on scroll; everything else in the "do not animate" list from v1 still holds.

**Why this satisfies "at least one of pin/parallax/stagger":** it ships all three (items 1, 2, 3), not just one, because the hero pin and the ghost-numeral parallax are cheap to combine (same `ScrollTrigger` region) and the Skills stagger is required anyway by the Layout & Composition restructure (individual tiles need individual reveal timing to read as individual tiles, not one block).
